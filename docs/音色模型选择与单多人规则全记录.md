# 音色模型选择与单多人规则全记录

日期：2026-06-17

本文记录本轮 session 中关于 BabelEcho 中文 TTS 音色、单人/多人播客、男女声区分、模型选择和最终部署规则的完整讨论与结论。这里的内容覆盖已尝试路线、被推翻的中间规则、最终代码规则、验证产物和后续注意事项。

## 1. 最终结论

最终运行规则已经改为：**只部署并默认使用 `CosyVoice-300M-SFT`**。

当前 BabelEcho MVP-1 TTS 默认配置和自动路由不再要求部署 `CosyVoice2-0.5B`。`CosyVoice2` 的 `zero_shot` / `cross_lingual` 样本只作为历史校准记录保留，不作为当前默认运行路径。

当前默认 voice：

```yaml
tts:
  provider: local_cli
  command: "tts-wrapper"
  voice: "sft_builtin_4role"
  cosyvoice_repo: "/path/to/CosyVoice"
  speed: 1.0
  output_format: "wav"
```

当前自动角色规则：

| 场景 | 当前模型 | voice role | SFT speaker id | 说明 |
| --- | --- | --- | --- | --- |
| 0/1 个 speaker，且没有显式性别标签 | `CosyVoice-300M-SFT` | `female_a` | `中文女` | 单人默认女声，避免回到旧模型 |
| 1 个 speaker，标签包含 `male` 或 `男` | `CosyVoice-300M-SFT` | `male_a` | `中文男` | 单人男声 |
| 1 个 speaker，标签包含 `female` 或 `女` | `CosyVoice-300M-SFT` | `female_a` | `中文女` | 单人女声 |
| 2+ 个 speaker | `CosyVoice-300M-SFT` | 首次出现顺序分配 | `中文女 / 中文男 / 英文女 / 英文男` | 多人稳定区分 |

多人首次出现顺序：

```text
第 1 个 speaker -> female_a -> 中文女
第 2 个 speaker -> male_a   -> 中文男
第 3 个 speaker -> female_b -> 英文女
第 4 个 speaker -> male_b   -> 英文男
第 5 个及以后 -> 循环复用上述四个角色
```

重要限制：

- 不做原主播 voice clone。
- 不做声纹识别。
- 不从音频或语义自动判断 speaker 性别。
- 只识别 `speaker` 标签中明确出现的 `male` / `男` / `female` / `女`。
- 如果 speaker 只是 `Alice`、`Bob` 这类名字，多人按首次出现顺序分配；单人默认 `female_a`。
- DeepSeek 只用于中文口播稿 adaptation，不参与 TTS 音色校准。多数音色实验未调用 DeepSeek。

## 2. 为什么最开始会有“多个音色”

用户最初的问题是：模型自带看起来只有两个音频文件，为什么能生成多个不同音色。

当时讨论的核心解释是：

- `CosyVoice2-0.5B` 的 `asset/zero_shot_prompt.wav` 和 `asset/cross_lingual_prompt.wav` 不是“只能生成两个固定声音”的严格开关。
- 对 `zero_shot` / `cross_lingual` 路线来说，`prompt_wav` 更像参考条件：模型会从参考音频中提取音色、节奏、说话方式等条件，再结合文本进行生成。
- 同一个模型在不同 `mode`、不同 `prompt_wav`、不同 `prompt_text`、不同 `speed`、不同采样随机性、不同后处理下，会产生听感差异。
- 这些差异不是稳定的“角色库”。它们更像参考音频和生成参数共同影响出来的结果。
- 所以可以得到多个听感不同的样本，但不等于模型有可靠的、多 speaker 可控音色集合。

这个解释后来直接影响了模型选择：`CosyVoice2` 可以用于单固定音色校准，但它不适合作为稳定的多人角色分配方案。

## 3. 情感是如何调整的

用户随后问“之前的情感是如何调整的”。当时的解释是：

`CosyVoice2` 这一路没有使用真正的情感控制器，也没有训练或修改模型权重。情感和播客语气主要通过以下手段间接影响：

- `mode=zero_shot`：使用 `zero_shot_prompt.wav` 和 prompt text。参考音频会带入一定说话风格。
- `mode=cross_lingual`：使用 `cross_lingual_prompt.wav`，让中文文本借用跨语言参考音频的音色和节奏条件。
- `inference_instruct2`：尝试用指令把语气压到更自然、克制、不夸张。
- `speed`：例如 1.0、0.95、0.90。速度越慢，播客感可能更稳，但也可能拖沓。
- 后处理：包括 pitch、EQ、loudnorm 等，用于让声音更亮、更低、更清楚或响度更一致。

当时得出的结论是：

- 这些方法可以调整“听感”和“情绪强弱”，但控制力度有限。
- 它们不是稳定的情感标签系统。
- 对播客来说，不希望情感过大，因此优先选择自然、克制、可长时间收听的声音。

## 5. 不继续围绕两个内置 wav 反复微调

当时讨论过：

- 5090D 上 `CosyVoice2` 目录只有两个内置参考音频：
  - `asset/zero_shot_prompt.wav`
  - `asset/cross_lingual_prompt.wav`
- 继续只围绕这两个 wav 调参数，收益有限。
- 下一步如果还要找新固定音色，应准备本地授权的男声/中性参考 wav，用同样的 `cross_lingual` 路线替换 `prompt_wav` 对比。
- 这类工作属于“固定音色扩展”，不是原主播 voice clone。

后来因为 `CosyVoice-300M-SFT` 能同时满足单男、单女、多人需求，这条“找本地授权 reference wav”的工作被降级为 deferred voice work，并且后续如果做，也应避免重新引入双模型部署。

### 7.2 为什么不能可靠地“男生样本出男声、女生样本出女声”

当时进一步解释：

- `CosyVoice2` 的 `cross_lingual` / `zero_shot` 不是严格的 speaker-id 选择器。
- 参考 wav 会影响音色，但模型不保证完整复制参考说话人的性别、年龄、共振峰和声线。
- 跨语言参考时尤其容易出现漂移：英文/德文/荷兰文参考音频用于中文生成，模型可能保留节奏或音色的一部分，但不保证稳定的男女边界。
- 多次调用会有生成随机性，但随机性主要带来细微差别，不会稳定产生可靠的“另一个男声”。
- 后处理可以让声音更低、更亮、更暗，但它只是改最终音频频谱，不会让模型内部形成真正的男声 speaker identity。

这也是后来寻找“权重模型”或带内置 speaker id 模型的原因。

## 9. 选择带内置 speaker id 的 300M SFT 模型

### 16.1 `src/babelecho/synthesize.py`

当前职责：

- 读取 `script/zh.json`。
- 计算 effective TTS config。
- 强制选择 `sft_builtin_4role`。
- 移除旧 CosyVoice2 单人参数：
  - `mode`
  - `prompt_text`
  - `prompt_wav`
- 为每个 segment 写入 `voice_role`。
- 在 `segments/manifest.json` 中记录 `tts_voice` 和每段 `voice_role`。

当前单人角色判断：

```text
speaker label 包含 female / woman / 女 -> female_a
speaker label 包含 male / man / 男 -> male_a
否则 -> female_a
```

当前多人角色判断：

```text
按 speaker 首次出现顺序分配 female_a / male_a / female_b / male_b
同名 speaker 复用同一角色
超过 4 个 speaker 循环复用
```

### 16.2 `src/babelecho/tts.py`

当前 `local_cli` 默认 voice：

```python
DEFAULT_LOCAL_CLI_VOICE = "sft_builtin_4role"
```

这意味着即使调用方没有写 `tts.voice`，底层 wrapper 命令也会带：

```text
--voice sft_builtin_4role
```

### 16.3 `tools/cosyvoice_tts_wrapper.py`

当前 wrapper 默认 voice：

```python
DEFAULT_VOICE = "sft_builtin_4role"
```

SFT 角色映射：

```python
SFT_ROLE_SPEAKERS = {
    "female_a": "中文女",
    "male_a": "中文男",
    "female_b": "英文女",
    "male_b": "英文男",
}
```

SFT 模型目录规则：

- 如果显式传 `--model-dir`，使用显式路径。
- 如果没有传 `--model-dir`，默认使用：

```text

```

- 对 `sft_builtin_4role`，wrapper 不会复用 `COSYVOICE_MODEL_DIR`。
- 这样即使 launcher 里旧环境变量还指向 `CosyVoice2-0.5B`，也不会把运行路径拉回旧模型。

## 19. 当前模型选择解释

### 19.1 为什么不用 CosyVoice2 做默认

原因：

- 单人默认曾经可用，但男女控制不可靠。
- 多人角色区分不稳定。
- 外部 reference wav 方案需要更多授权素材。
- 男声 reference 在中文合成里可能听起来仍像女声。
- 多次调用同一个 zero-shot reference 只产生小差别，不构成稳定角色。
- 如果保留 CosyVoice2 + 300M 双模型，会增加 5090D 部署复杂度。
- 用户最终目标明确为只用 300M。

### 19.2 为什么用 CosyVoice-300M-SFT

原因：

- 内置 speaker id 更稳定。
- 已验证单男、单女、多人都能应付。
- 不需要额外 reference wav。
- 不需要 voice clone。
- 不需要修改模型权重或 `spk2info.pt`。
- 更适合 MVP-1 的固定播客角色需求。

### 19.3 为什么仍保留 wrapper 里的 `default-zh`

低层 wrapper 仍保留 `default-zh` 支持，主要是为了：

- 历史 smoke test 和手动对比能继续跑。
- 保留 `zero_shot` / `cross_lingual` 诊断能力。
- 不破坏旧命令。

但项目级默认已经不依赖它：

- `synthesize` 会覆盖旧 `default-zh` 为 `sft_builtin_4role`。
- `local_cli` 默认 voice 是 `sft_builtin_4role`。
- wrapper 默认 voice 是 `sft_builtin_4role`。

## 21. 不能误解的点

### 21.1 “单人默认 female_a”不是声纹判断

如果脚本只有一个 speaker 且名字是 `Solo`、`Host` 或为空，系统不会判断真实性别，直接使用 `female_a`。

### 21.2 `MaleHost` / `FemaleHost` 是标签约定

当前只靠 speaker label 中的字符串：

- `MaleHost` 命中 `male` -> `male_a`
- `FemaleHost` 命中 `female` -> `female_a`
- `男主持` 命中 `男` -> `male_a`
- `女主持` 命中 `女` -> `female_a`

如果真实 transcript 只有人名，不会自动推断性别。

### 21.3 多人不按名字性别分配

多人目前按首次出现顺序分配，不按名字判断：

```text
Alice -> female_a
Bob   -> male_a
Carol -> female_b
Dave  -> male_b
Alice -> female_a
```

这保证稳定和简单，但不保证真实 speaker 性别匹配。

### 21.4 `英文女` / `英文男` 是角色，不是英文输出

`female_b` 和 `male_b` 使用 `CosyVoice-300M-SFT` 的 `英文女` / `英文男` speaker id，但输入文本仍是中文。选择它们是为了提高角色区分度，不是为了生成英文。

### 21.5 这不是 voice clone

全程没有用原播客主播音频训练或克隆声音。`CosyVoice-300M-SFT` 用的是模型内置 speaker id。

## 22. 后续建议

后续与音色相关的工作应遵守以下边界：

- 不重新引入双模型默认部署，除非用户明确改变目标。
- 优先继续验证 `CosyVoice-300M-SFT` 在真实 RSS 多 speaker 节目里的听感。
- 如果真实节目 speaker 顺序导致角色分配不合理，可以增加人工 speaker 修正文件或每节目 speaker 映射配置。
- 如果要扩展更多固定音色，优先找同样能单模型部署的方案，而不是回到 `CosyVoice2 prompt_wav` 双模型路线。
- 如果用户以后要求真实主播音色，再单独讨论 voice clone；当前 MVP-1 明确不做。
