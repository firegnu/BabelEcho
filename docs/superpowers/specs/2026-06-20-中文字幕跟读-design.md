# 中文字幕跟随播放设计

日期：2026-06-20

## 目标

让前端播放中文音频时，下方的中文脚本能随播放进度实时跟随：高亮当前正在播放的段落、自动平滑滚动到该段，并支持点击任意段落跳转到该段音频位置（双向）。覆盖两种中文展示视图：

- podcast 中文脚本视图（分段卡片，`route != article_reading`）。
- 文章朗读阅读视图（正文 + 小标题段，`route == article_reading`）。

实现的前提是给中文段落补上**段级时间戳**。中文 MP3 是逐段 TTS 合成 wav 再顺序拼接得到的，原始英文 transcript 的时间戳（`transcript.en.json` 的 `start_ms`）是英文原音时间轴，与中文 MP3 时间轴不对齐，**不能复用**。因此中文时间戳必须从 TTS 合成结果重新推导。

## 非目标

- 不做逐字 / 字级高亮（没有字级时间戳，也不需要）。
- 不给英文原文 tab 做跟读（英文是原音时间轴，与中文 MP3 不对齐）。
- 不给已发布的旧集补时间戳、不做数据迁移；旧集前端自动降级为现有静态行为。
- 不引入前端 JS 测试框架（如 Playwright）；保持纯静态、无构建、无依赖。
- 不改任何来源路线（YouTube / 单 URL / RSS / iTunes / episode page / 文章 / audio-first），不改 `synthesize`，不改英文 `transcript.en.json`。

## 背景：三路汇聚与现状事实

- 三条编排（transcript-first 主线、article、audio-first）在 `adapt` 之后完全合流，共用同一套 `synthesize_segments`（`synthesize.py`）、`assemble_audio`（`audio.py`）、`publish_episode`（`publish.py`）。因此本功能只改这条统一后段，所有来源路线一次受益，无需按路线分别改动。
- `assemble_audio` 用 ffmpeg concat demuxer **纯顺序拼接各段 wav、无段间 gap/静音**，再整体编码为 MP3。所以最终 MP3 时间轴 = 各段 wav 时长按 `manifest["segments"]` 顺序累加。
- `publish_episode` 目前用 `shutil.copy2(script/zh.json → transcript.zh.json)`，即前端读的 `transcript.zh.json` 就是 run-local `script/zh.json` 的纯拷贝。`transcript.zh.json` 段字段当前为 `id / source_segment_ids / speaker / text`，无时间戳。
- fixture TTS provider（`tts.py::write_silent_wav`）生成的是**有真实时长的**静音 wav。因此整条时间戳链路与前端跟读，本机 fixture 即可端到端验证，不依赖 5090D 真跑 TTS。

## 架构与数据流

时间戳来源选定**方案 A：在 `assemble` 累加、在 `publish` 注入**。

```text
synthesize: 逐段生成 wav，写 manifest（段: id, audio_path, speaker, text, voice_role ...）
   │
assemble:  遍历 manifest 各段 → 用标准库 wave 读每段 wav 时长(nframes/framerate)
   │        → 按顺序累加 start_ms/end_ms/duration_ms → 回写 manifest 各段
   │        → ffmpeg concat 拼 MP3（行为不变）
   │
publish:   读 script/zh.json + 按 id merge manifest 段级时间戳 → 写 transcript.zh.json
   │        （不再纯 copy；script/zh.json 保持 adapt 纯产物不变）
   │
前端:      transcript.zh.json 段含 start_ms/end_ms → 渲染时挂 data-start/data-end
            → ontimeupdate 二分定位当前段 → 高亮 + 平滑滚动；点击段 → seek
```

为什么算在 `assemble`：拼接顺序的唯一权威点就在 `assemble`，在那里累加最贴近真实 MP3 时间轴；用 Python 标准库 `wave` 读 header，零新依赖、零子进程。`synthesize` 与各来源路线不动。

## 后端改动

### 1. `audio.py::assemble_audio`

在现有遍历 `manifest["segments"]` 拼 concat 的循环中，对每段 wav 读时长并累加：

- 用标准库 `wave` 打开每段 wav，`duration = nframes / framerate`。
- 维护累计游标 `cursor_ms`，每段写入 `start_ms = round(cursor_ms)`、`end_ms = round(cursor_ms + duration*1000)`、`duration_ms = end_ms - start_ms`，然后 `cursor_ms += duration*1000`。
- 把这三个字段写回该段，结束后 `write_json` 覆盖 `manifest.json`。
- ffmpeg 拼接命令与返回值不变。

边界：段 wav 缺失或不可读时按现有失败语义抛错（assemble 本就依赖这些 wav 存在）。

### 2. `publish.py::publish_episode`

将 `transcript.zh.json` 的生成从「纯 copy」改为「读 `script/zh.json` + merge manifest 时间戳」：

- 读 `manifest.json`，建 `id -> {start_ms, end_ms, duration_ms}` 映射。
- 读 `script/zh.json`，对每段按 `id` 注入时间戳字段后写出 `transcript.zh.json`。
- 若某段在 manifest 找不到（理论上不应发生），该段不带时间戳，保持可降级。
- `script/zh.json` 本身不改；`transcript.en.json` 不改。

### 兼容判定

前端启用跟读的判据是「`transcript.zh.json` 段里是否存在 `start_ms`」，因此**不需要**在 `artifact.json` 新增开关字段。旧集无 `start_ms` → 前端静态展示；新集有 → 启用跟读。

## 前端改动（`frontend/`）

### `js/app.js`

- `scriptHtml`（podcast 分段视图）与文章朗读 reading 视图渲染时，给每段 DOM 挂 `data-start`/`data-end`（ms）。文章视图按 `ep.zh` 段顺序对应正文段/小标题段。
- 新增「当前段定位」：在 `wirePlayer` 的 `ontimeupdate`（`paint`）里按 `audio.currentTime*1000` 对段时间戳二分查找当前段；给当前段加 `.active`，移除上一段的。
- 自动滚动：当前段变化时 `scrollIntoView({behavior:'smooth', block:'center'})`，**仅当用户未在手动滚动时**。检测用户手动滚动则暂停自动跟随一小段时间（如 ~3s 无滚动后恢复），避免回看被强行拉回。
- 点击跳转：点击带 `data-start` 的段 → `audio.currentTime = start/1000`。
- 仅当本集段含时间戳时启用以上行为；否则维持现状（静态渲染、无高亮/滚动），不报错。
- tab 切换会重渲染 panel：切回中文脚本/正文 tab 后需重新挂事件与恢复当前 `.active`。

### `css/app.css`

- 新增 `.seg.active` / `.reading-p.active` / `.reading-h.active` 高亮样式（用现有 `--accent` / `--accent-2` token，podcast 用 accent、文章用 accent-2），与暗色主题一致。
- 段落加 `cursor:pointer` 提示可点击跳转（仅有时间戳时）。

### 文档

实现后同步 `docs/前端Artifact契约与只读界面说明.md`：说明 `transcript.zh.json` 段在带时间戳时含 `start_ms/end_ms/duration_ms`，前端据此启用中文跟读。

## 时间戳精度

时间戳是「各段 TTS wav 时长累加」，与最终 MP3 实际时间轴存在 MP3 编码帧对齐级别的毫秒漂移（全程累计通常 <100ms）。段级跟读只需判断「当前在第几段」，对该量级漂移无感，可接受。不追求逐字毫秒精度。

## 测试策略

后端单元测试（TDD，先写测试看到预期失败再实现）：

- `assemble`：构造多段已知时长的 wav（标准库 `wave` 写），跑 `assemble_audio` 后断言 manifest 各段 `start_ms/end_ms/duration_ms` 按顺序正确累加、首段 `start_ms=0`、相邻段首尾相接。
- `publish`：跑 publish 后断言 `transcript.zh.json` 各段含正确的 `start_ms/end_ms`，且与 manifest 一致；`script/zh.json` 不被改动；`transcript.en.json` 不变。
- fixture 端到端（`test_end_to_end_fixture`）：跑通到 publish 后，断言 `transcript.zh.json` 段带时间戳（fixture 静音 wav 有真实时长，本机闭环）。
